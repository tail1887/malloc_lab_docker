## 1. 과제가 요구하는 것

### 1.1 목표

동적 메모리 할당기를 직접 구현합니다. 사용자(드라이버)는 **`malloc` / `free` / `realloc`과 같은 요청**을 연속으로 보내고, 당신의 구현은:

- **정확성**: 잘못된 포인터, 겹치는 블록, 정렬 위반 등이 없어야 함
- **공간 효율**: 힙을 얼마나 낭비 없이 쓰는지(이용률)
- **시간 효율**: 요청을 얼마나 빨리 처리하는지(처리량)

을 만족해야 합니다.

### 1.2 수정하는 파일

- **`malloc-lab/mm.c`**: **유일하게 수정·제출 대상이 되는 구현 파일**이라고 가정하면 됩니다.
- **`mm.h`**, **`memlib.c`** 등은 인터페이스·시뮬레이터이므로 일반적으로 수정하지 않습니다.

### 1.3 구현해야 하는 함수 (`mm.h`)

| 함수 | 역할 |
|------|------|
| `int mm_init(void)` | 할당기 상태 초기화. 실패 시 음수, 성공 시 0 등 과제 예시에 맞게 |
| `void *mm_malloc(size_t size)` | `size` 바이트 이상의 **페이로드**를 담을 수 있는 블록 할당. 실패 시 `NULL` |
| `void mm_free(void *ptr)` | 이전에 `mm_malloc` / `mm_realloc`이 돌려준 포인터 해제 |
| `void *mm_realloc(void *ptr, size_t size)` | 기존 블록 크기 변경. `ptr == NULL`이면 `malloc`과 유사, `size == 0` 처리는 구현/스펙 확인 |

### 1.4 정렬

`config.h`에서 **`ALIGNMENT`는 8**입니다.  
**반환하는 페이로드 포인터**는 항상 **8바이트 정렬**되어 있어야 합니다.

### 1.5 팀 정보

`mm.c`의 `team_t team`에 팀명, 이름, 이메일(또는 과제에서 요구하는 ID)을 채웁니다.

## 3. 가상 힙 모델: memlib

실제 OS의 `sbrk` 대신, 이 과제는 **`memlib`**으로 **고정 크기 배열 위의 힙**을 시뮬레이션합니다.

### 3.1 중요 상수 (`config.h`)

- **`MAX_HEAP`**: 기본 **20MB**. `mem_sbrk`로 늘릴 수 있는 상한에 해당합니다.

### 3.2 제공 함수 (`memlib.h`)

| 함수 | 의미 |
|------|------|
| `void *mem_sbrk(int incr)` | 힙을 `incr` 바이트만큼 **확장**. 반환값은 **이번에 새로 예약된 구간의 시작 주소**(기존 brk). 실패 시 `(void *)-1` |
| `void mem_reset_brk(void)` | brk를 힙 시작으로 되돌림 → **힙 비우기** |
| `void *mem_heap_lo(void)` | 힙 첫 바이트 주소 |
| `void *mem_heap_hi(void)` | 힙 **마지막** 바이트 주소 |
| `size_t mem_heapsize(void)` | 현재 힙 크기 |
| `size_t mem_pagesize(void)` | 페이지 크기(참고용) |

### 3.3 구현 시 주의

- **`mem_sbrk`는 힙을 줄이지 않습니다** (`incr < 0`이면 실패).
- 할당기는 **오직 이 API로만** 힙 영역을 늘립니다.
- 블록 **헤더/푸터/패딩**도 전부 이 힙 안에 들어가므로, **페이로드만큼만** 크기를 잡으면 안 되고 **메타데이터 + 정렬**을 포함해 계산해야 합니다.

## 4. 드라이버 mdriver와 트레이스 파일

### 4.1 빌드

```bash
cd malloc-lab
make
```

실행 파일: **`mdriver`**

### 4.2 트레이스 한 줄 형식

`.rep` 파일은 대략 다음 구조를 가집니다(파일마다 동일 패턴).

- 앞쪽 몇 줄: 메타데이터(힙 제안 크기, id 개수, 연산 개수, 가중치 등) — **드라이버가 파싱**
- 이후 각 줄: 한 번의 할당기 요청

연산 예시:

- `a <id> <size>` — **할당**. 드라이버는 반환 포인터를 `id`에 저장
- `f <id>` — **해제**. 해당 `id`에 저장된 포인터로 `free`
- `r <id> <size>` — **realloc**

작은 예제로 **`short1-bal.rep`**를 열어보면 순서 감을 잡기 좋습니다.

### 4.3 자주 쓰는 명령

```bash
# 도움말
./mdriver -h

# 짧은 트레이스 하나, 자세한 로그
./mdriver -V -f short1-bal.rep

# 기본 트레이스 전체 (config.h의 DEFAULT_TRACEFILES)
./mdriver

# 트레이스 디렉터리 지정
./mdriver -t ./traces/
```

`-V`(verbose)는 **각 연산이 어떻게 호출되는지** 보여 주므로, 초반 학습·디버깅에 매우 유용합니다.

### 4.4 드라이버가 검사하는 것(개념)

- 반환 포인터가 **힙 범위** 안인지
- **8바이트 정렬**인지
- 서로 다른 살아 있는 블록의 **페이로드 구간이 겹치지 않는지**
- `free`에 넘긴 포인터가 **유효한 할당 블록의 페이로드 시작**인지
- `realloc` / `free` 후 **잘못된 포인터 사용**이 없는지

정확성 검사를 통과해야 **이용률·속도** 통계가 의미 있습니다.

## 5. 채점에 가까운 지표: config.h

### 5.1 성능 지수(Performance index)

드라이버는 트레이스별로 대략 다음을 측정합니다.

- **Utilization(이용률)**: 대략 “살아 있는 페이로드 합 / 힙에서 쓴 크기”에 가까운 개념(드라이버 구현 기준).
- **Throughput(처리량)**: 초당 연산 수에 가까운 개념.

`config.h`에서:

- **`UTIL_WEIGHT` = 0.6** → **이용률이 점수에 더 큰 비중**
- **처리량 비중** ≈ **0.4**
- **`AVG_LIBC_THRUPUT` (600 Kops/sec)** 을 넘기면 처리량 쪽 이득이 **더 이상 늘지 않도록** 상한이 잡혀 있음 → **극단적으로 빠르지만 낭비가 큰 할당기**는 비효율적

### 5.2 실전에서의 시사점

1. **먼저 정확성** (모든 기본 트레이스 통과)  
2. 그다음 **내부 단편화·외부 단편화 줄이기** (free 리스트, 병합, 분할, 적절한 자료구조)  
3. 처리량은 **불필요한 탐색·복잡한 연산 줄이기**로 맞추되, **이용률과 트레이드오프**를 보며 조정

## 6. 시작 코드(mm.c)가 하는 일

제공되는 `mm.c`는 **“naive”** 에 가깝습니다.

- **`mm_malloc`**: `mem_sbrk`로 크기만큼 힙을 밀고, 앞부분에 `size_t` 하나를 저장한 뒤 그 **다음**을 페이로드로 반환
- **`mm_free`**: **아무 것도 안 함** → 메모리 재사용 없음, 이용률 매우 나쁨
- **`mm_realloc`**: 새로 `malloc` → `memcpy` → `free` 하는 단순 형태

이 상태로는 **작은 트레이스**도 어떤 설정에서는 문제가 생길 수 있고, **전체 트레이스**에서는 이용률·속도 모두 좋지 않습니다.  
**여기서부터 자신만의 블록 구조 + free 관리 + 병합**을 구현하는 것이 과제 본체입니다.

## 7. 반드시 알아야 할 개념

### 7.1 블록과 페이로드

- **페이로드**: 사용자가 실제로 데이터를 넣는 영역. `mm_malloc`이 반환하는 포인터가 가리키는 곳.
- **블록**: 페이로드 + **헤더(및 경우에 따라 푸터)** 등 **할당기가 관리하는 전체 덩어리**.

크기를 잡을 때 **“페이로드 크기”와 “전체 블록 크기”** 를 혼동하지 않도록 항상 구분합니다.

### 7.2 정렬

요청 크기 `size`에 대해, **페이로드 시작 주소**가 8의 배수가 되도록 **내부 블록 크기와 헤더 배치**를 설계합니다.  
(구현마다 “헤더 직후가 곧 페이로드” 또는 “페이로드 전에 정렬 패딩” 등 방식이 달라질 수 있습니다.)

### 7.3 경계 태그(Epilogue / Prologue)

많은 구현에서:

- 힙 **맨 앞/맨 끝**에 **더미 블록** 또는 **특수한 0 크기 에필로그**를 두어  
- **이전/다음 물리적 이웃 블록**을 찾고 **병합(coalescing)** 할 때 경계 처리를 단순화합니다.

### 7.4 할당 블록 vs 가용(free) 블록

- **Allocated**: 사용자에게 넘긴 상태. 페이로드만 사용자가 건드려도 됨(헤더는 할당기 전용).
- **Free**: 재사용 가능. 보통 **free 리스트**로 연결하거나, **묵시적(implicit) 순회**로 찾습니다.

### 7.5 병합(Coalescing)

`free` 시 **앞뒤 이웃이 모두 free**이면 **하나의 큰 free 블록**으로 합치는 것.  
**외부 단편화**를 줄이는 핵심입니다.

### 7.6 분할(Splitting)

할당 시 free 블록이 요청보다 **훨씬 크면**, 앞부분만 할당하고 **나머지는 작은 free 블록**으로 남깁니다.  
안 하면 **내부 단편화**가 커질 수 있습니다.

### 7.7 탐색 정책

free 블록을 찾을 때:

- **First fit**: 처음 맞는 것
- **Next fit**: 이전 탐색 위치부터
- **Best fit**: 충분한 것 중 최소 크기(구현·비용 주의)

초기에는 **first fit**으로 정확성부터 잡고, 이후 튜닝하는 방식을 권장합니다.

### 7.8 realloc의 의미

- **크기 축소**: 일부만 복사 후 이전 블록 처리
- **크기 확대**  
  - **같은 자리에서 늘릴 수 있으면** (뒤가 free이거나 새 sbrk 영역) **복사 최소화**  
  - 아니면 새 블록 할당 후 복사·해제

처음에는 **항상 malloc + memcpy + free**로 두고, 트레이스 통과 후 최적화해도 됩니다.

## 8. 설계 선택지 개요

아래는 교재에서 흔히 소개되는 스펙트럼입니다. **하나를 정해 끝까지 구현**한 뒤 개선하는 편이 좋습니다.

| 방식 | 장점 | 단점 |
|------|------|------|
| **Implicit free list** | 구현 단순, 포인터 저장 공간 적음 | free 탐색이 O(힙 블록 수) |
| **Explicit free list** | free만 순회, 상대적으로 빠름 | free 블록에 succ/pred 포인터 공간 필요 |
| **Segregated free lists** | 크기별 리스트로 탐색 범위 축소 | 구현·튜닝 복잡 |
| **Binary tree 등 (고급)** | best-fit에 가까운 탐색 | Malloc Lab 범위에서는 부담 클 수 있음 |

대부분의 학습자는 **명시적 리스트 + 경계 태그 + 병합** 조합으로 목표 점수까지 갈 수 있습니다.

## 9. 권장 구현 순서

### 단계 A — 종이 설계

1. 그림으로 **한 블록의 비트 레이아웃**(할당 비트, 크기 필드 크기, 최소 블록 크기).
2. **최소 블록 크기**: free 리스트에 넣을 때 들어가야 하는 **포인터/메타**를 두기 위한 하한.
3. **포인터 역참조**: `free(ptr)`에서 `ptr`에서 **헤더 주소**를 어떻게 구하는지 한 줄로 씀.

### 단계 B — mm_init

- 전역 변수(힙 리스트 헤드, 탐색 커서 등) 초기화.
- 필요 시 **프롤로그/첫 블록** 구성.
- **`mm_init` 실패 시 -1** 등, 드라이버가 기대하는 규약 확인.

### 단계 C — mm_malloc

1. 요청 `size` → **정렬된 페이로드 크기** 결정.
2. **전체 블록 크기** = 메타데이터 + 페이로드 + (필요 시 패딩).
3. free 리스트에서 탐색 → 있으면 **할당 처리 + split 여부** 결정.
4. 없으면 `mem_sbrk`로 새 블록.
5. 헤더에 크기/할당 비트 기록, **페이로드 포인터 반환**.

### 단계 D — mm_free

1. `ptr`에서 헤더 찾기.
2. allocated → free로 표시.
3. **앞/뒤 이웃과 병합**.
4. explicit list면 **리스트에 삽입**(앞/뒤 삽입 정책 정하기).

### 단계 E — mm_realloc

1. `ptr == NULL` → `mm_malloc(size)`.
2. `size == 0` — 과제/ libc 호환 관행 확인(보통 free 후 NULL 또는 작은 할당 등).
3. 그 외: 우선 **단순 malloc/copy/free**로 통과.
4. 이후 **in-place 확장**, **다음 free와 병합 후 확장** 등 적용.

### 단계 F — 검증 루프

1. `short1-bal.rep`, `short2-bal.rep`
2. `./mdriver` 전체
3. 이용률·처리량 보며 **split, 탐색 정책, 최소 블록** 조정

## 10. mm_init이 여러 번 호출된다

`mdriver.c` 흐름상, 트레이스를 돌리기 전에 **`mem_reset_brk()`** 로 힙을 비운 뒤 **`mm_init()`** 을 호출합니다.  
즉, **각 트레이스(또는 각 평가 단계)마다 빈 힙에서 다시 시작**한다고 보면 됩니다.

**반드시:**

- 이전 실행의 **전역 리스트 포인터·힙 상태를 남기지 말 것**
- 매번 **일관된 초기 구조**로 힙을 세팅할 것

`mm_init` 안에서만 “최초 한 번만” 하는 전역 설정이 있다면 버그로 이어지기 쉽습니다.

## 11. 디버깅 전략

1. **`-V`와 짧은 트레이스**로 실패 연산 번호를 좁힌다.
2. **assert**:  
   - 블록 크기가 최소 크기 이상인지  
   - free 리스트에 중복 삽입이 없는지  
   - 할당 블록의 할당 비트가 맞는지  
3. **힙 덤프 함수**를 잠깐 만들어, `mm_malloc`/`mm_free` 후에 블록 목록을 `printf` (제출 전 제거 또는 매크로로 끄기).
4. VS Code **F5**: 프로젝트에 `malloc-lab` 기준 디버그 설정이 있으면 `mdriver`를 인자와 함께 실행할 수 있음.
5. **Valgrind**: Dev Container에 `valgrind`가 설치되어 있으면(이 Docker 이미지는 포함) 메모리 오류 추적에 유리.

## 12. 자주 나는 실수

- **페이로드 크기와 블록 전체 크기 혼동** → 힙 침범 또는 정렬 깨짐.
- **포인터 역산 오류** (`char *` 산술, 캐스팅) → 잘못된 free.
- **병합 시 이중 삽입/리스트 깨짐** → explicit list에서 특히 빈번.
- **split 후 최소 블록 미만**으로 쪼개기 → 다음 할당 시 깨짐.
- **`mm_init` 재호출** 대비 안 함 → 두 번째 트레이스부터 오동작.
- **realloc**에서 이전 블록 크기를 잘못 읽어 **memcpy 범위 초과**.
